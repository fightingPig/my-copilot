# High Risk And Sensitive Rulebook

## 使用原则

这份文件专门沉淀“写前必须拦截”的高危与敏感规则。

使用方式：

1. 每条规则必须有稳定编号。
2. `PreToolUse` 命中后，返回信息必须带规则编号。
3. 人工介入时，优先依据规则编号判断是继续拦截、降级为写后反馈，还是补充规则说明。
4. 新增案例时，不只记录“禁止什么”，还要记录“为什么危险”“推荐替代方案”“真实案例”。

---

## HR-001 禁止使用 System.out/System.err 打印业务日志

- 类型：高危红线
- 默认动作：写前拦截
- 风险说明：
  `System.out/System.err` 容易绕过统一日志体系，导致日志不可检索、不可分级、不可审计。
- 推荐替代：
  使用 SLF4J。
- 典型替代示例：
  `log.info("order created, orderId={}", orderId);`
- 备注：
  若是临时本地调试代码，也不应进入正式代码写入流程。

## HR-002 禁止使用 Executors 快捷工厂创建线程池

- 类型：高危红线
- 默认动作：写前拦截
- 风险说明：
  `Executors` 快捷工厂容易隐藏队列或线程增长风险，在线上可能引发 OOM 或排障困难。
- 推荐替代：
  手动构造 `ThreadPoolExecutor`，明确核心参数和线程名。
- 典型替代示例：
  显式设置核心线程数、最大线程数、阻塞队列、拒绝策略和线程名前缀。

## HR-003 禁止硬编码密码、Token、AK/SK、密钥

- 类型：敏感信息
- 默认动作：写前拦截
- 风险说明：
  敏感信息一旦落盘，就可能被误提交、被复制到日志或被二次传播，代价远高于普通规范问题。
- 推荐替代：
  从配置中心、环境变量或密钥管理系统读取。
- 人工介入建议：
  若 AI 反复生成类似内容，优先加强提示词或补充上下文，而不是放宽规则。

## HR-004 禁止明显未参数化的 SQL 字符串拼接

- 类型：高危红线
- 默认动作：写前拦截
- 风险说明：
  直接字符串拼接容易引入 SQL 注入，并使 SQL 行为依赖输入拼接细节。
- 推荐替代：
  使用参数化查询、MyBatis `#{}`、白名单字段映射。
- 人工介入建议：
  如果只是误判的普通字符串拼接，应评估是否降级为写后反馈规则。

## 沉淀模板

新增规则时，按以下模板追加：

```markdown
## HR-XXX 规则标题

- 类型：高危红线 / 敏感信息 / 待观察
- 默认动作：写前拦截 / 写后反馈 / 人工确认
- 风险说明：
  说明为什么危险、为什么不适合先落盘。
- 推荐替代：
  说明推荐实现方式。
- 真实案例：
  记录一次真实出现的问题、误报或修复经验。
- 规则决策：
  说明为什么当前定为拦截或反馈。
```

## 已知经验

1. 低歧义、高危害、落盘代价大的内容，优先放进这份文件并在 `PreToolUse` 拦截。
2. 依赖上下文判断、误报成本高的问题，不要轻易提升到这里，应优先保留在写后反馈链路。
3. 如果某条规则长期误报，应先补案例和边界说明，再决定是否降级。
